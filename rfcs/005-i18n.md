# RFC-005: Type-safe i18n with locale context

Status: accepted
Date: 2026-04-10

## Context

An accounting-focused design system needs locale-aware formatting for numbers, currencies, dates, and UI strings. Sage Carbon's approach (type-safe locale interface + context provider) is proven at scale.

## Decision

- `MizuLocale` interface defines all translatable strings and formatting functions
- Locale files export concrete implementations (en-US, en-GB, more to come)
- `I18nProvider` wraps the app and provides locale via React context
- `useLocale()` hook gives components access to the current locale
- Formatting functions use `Intl.NumberFormat`, `Intl.DateTimeFormat`, and `Intl.RelativeTimeFormat` — no external date/number library needed

## Alternatives considered

- **react-intl / i18next**: Full i18n frameworks with message catalogs, pluralization engines, extraction tooling. Overkill for a component library — we only need component-level strings and formatters, not app-level translation.
- **No i18n, let consumers handle it**: Would mean hardcoded English strings in components (empty states, pagination labels, ARIA text). Sage learned this doesn't scale.

## Consequences

- Components can use `useLocale()` for any user-facing string
- Consumers provide their locale via `<I18nProvider locale={enGB}>`
- Default is en-US if no provider is present (zero-config for US users)
- Adding a new locale = implementing the `MizuLocale` interface
- Trade-off: adds a React context lookup to every component that uses locale
